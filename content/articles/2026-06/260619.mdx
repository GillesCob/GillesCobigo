## Deux types de données, deux outils différents

Ce matin j’ai branché le dashboard de Cerithe sur le backend. Premier vrai appel API depuis le frontend, premier affichage de données réelles.

Et en le faisant j’ai compris quelque chose que j’avais déjà codé sans vraiment saisir toute les subtilités. Dans une app React, toutes les données ne se gèrent pas de la même façon. Les données locales, l’état d’un formulaire ou un menu ouvert, vivent dans Zustand. Les données serveur, la liste de mes biens, un profil utilisateur, ont un cycle de vie différent. Elles viennent de l’extérieur, elles peuvent être périmées, elles ont besoin d’être rechargées. C’est exactement ce que TanStack Query gère.

## Ce que useQuery fait à ta place

Sans TanStack Query, récupérer une liste depuis une API demande un useEffect qui fetch au montage, un useState pour stocker le résultat, un autre pour le loading, un try/catch pour les erreurs. Quatre mécanismes pour faire une chose simple.

Avec useQuery, c’est une clé unique et une fonction qui fetch. TanStack s’occupe du cache, du loading, des erreurs, de la revalidation.

## Le hook useProperties

Même logique que côté backend : on ne met pas la logique directement dans le composant. Le composant consomme un hook, le hook wrap le useQuery, le service fait l’appel API.

Renommer data en properties c’est un détail qui change tout à la lecture. Dans le dashboard on destructure properties et pas data. Six mois plus tard tu sais ce que c’est sans chercher.
![Légende de mon image](/imagesArticles/2026/06/19/2.png)

## Le loader avant tout

Avant d’afficher quoi que ce soit, deux vérifications. Si isPending est true, on affiche un loader. Si isError est true, on affiche un message d’erreur. Ensuite seulement on fait le map sur les données.

C’est une règle que j’avais notée comme point à travailler : loader systématique avant tout état vide. Aujourd’hui je l’ai appliquée sur Cerithe. Le résultat c’est un dashboard qui affiche l’état vide proprement quand l’utilisateur n’a pas encore de bien, pas une page blanche.

## Le IProperty et le problème de la duplication des types

Pour typer correctement le map sur les biens, j’ai créé une interface IProperty côté frontend. Même champs que le modèle Prisma côté backend, mais définis indépendamment.

C’est une vraie limitation du monorepo sans npm workspaces. Si je modifie le modèle backend, je dois penser à mettre à jour le type frontend. C’est de la dette technique, je l’assume pour le MVP. La solution propre c’est un package shared/types partagé entre les deux, ou tRPC qui gère ça nativement. A prévoir après le 17 juillet.

## Où on en est

Le dashboard affiche l’état vide proprement. La prochaine étape c’est le formulaire de création de bien, puis l’affichage de la liste quand les biens existent.
